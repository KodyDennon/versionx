# 11 — Production-Grade Version Roadmap

## Scope
Concrete release plan from 0.1.0 through 1.0 (GA) and beyond. Each 0.x is **production-grade for its scope** — shippable, signed, tested, documented — not a preview. The cumulative wedge (cross-repo atomic polyglot release with AI-as-client) only fully lands at 0.7–0.9; earlier versions deliver discrete slices that stand on their own.

## Contract
After reading this file you should be able to: tell what goes in each release, what gates each release must pass, and which demo each release unlocks.

---

## Philosophy

1. **Every 0.x ships to production.** Not alphas. Real binaries on GitHub Releases, signed, distributed, used by real people.
2. **Every 0.x has a 30-second demo.** If you can't make someone want it in 30 seconds, the scope was wrong.
3. **Every 0.x is incrementally adoptable.** A user on 0.3 should feel zero regression upgrading to 0.4.
4. **No breaking changes without a schema version bump.** `versionx.toml` + `versionx.lock` + RPC + MCP all carry `schema_version`.
5. **0.x semver is loose; 1.0 is strict.** Minor bumps pre-1.0 may introduce breakage with migration paths (`versionx migrate`). Post-1.0, full semver.
6. **Quality gates (§11) must all pass before tagging.** No shipping on "it probably works."

---

## Release gates (apply to every 0.x and the 1.0)

Before tagging any release, **all** of these must be true:

### Binary quality
- [ ] All three platforms build green in CI (Linux x86_64 glibc + musl + aarch64, macOS x86_64 + aarch64, Windows x86_64 + aarch64).
- [ ] Binary signed (macOS notarized via `rcodesign`; Windows Authenticode-signed via Azure Key Vault + AzureSignTool).
- [ ] Sigstore attestations attached to GitHub Release artifacts.
- [ ] `versionx --version` matches the tag; `versionx self-check` passes on all platforms.

### Test coverage
- [ ] Unit tests ≥80% line coverage on `versionx-core` and all shipped adapters.
- [ ] Fixture tests for every supported tool version (see matrix per release).
- [ ] E2E test battery green: install → init → sync → (feature under test) → cleanup.
- [ ] Property tests: any plan applied to its source state, re-planned, produces an empty plan (idempotence).

### Performance budgets
- [ ] Shim: `<1ms warm`, `<5ms cold` on all platforms (benchmark as CI gate).
- [ ] `versionx status` in a clean repo: `<200ms`.
- [ ] `versionx sync` on a 200-package pnpm monorepo (no changes): `<2s`.
- [ ] Daemon IPC round-trip: `<5ms` p99 local.

### Distribution
- [ ] GitHub Release published with release notes generated by versionx itself (dogfood).
- [ ] Homebrew tap updated (`acme/versionx`).
- [ ] Scoop manifest updated.
- [ ] curl|sh installer tested on fresh Ubuntu, macOS, and Windows runners.
- [ ] Cargo crate published (`cargo install versionx`).
- [ ] npm shim package published (`npm i -g versionx`).
- [ ] PyPI shim package published (`pip install versionx`).
- [ ] `versionx self update` successfully updates from the previous release on all platforms.

### Docs
- [ ] `docs/` pages for every new subcommand.
- [ ] Changelog updated (dogfood the release engine).
- [ ] Migration guide from previous version if schema bumped.
- [ ] JSON schema for `versionx.toml` published at `versionx.dev/schema/vN.json`.
- [ ] `versionx --help-json` output validates against committed schema.

### Zero-regression check
- [ ] The previous version's full demo script runs unchanged on the new binary.
- [ ] No open P0 or P1 bugs at tag time.

## 0.1.0 — Node-only foundation (2 months)

**Target date: M+2**. First production release.

### Scope
Rust workspace + CI + Node-only single-repo flow.

**Crates shipped:**
- `versionx-core` (skeleton + sync pipeline)
- `versionx-cli` (`versionx` binary via clap v4)
- `versionx-config` (toml_edit-based, schema v1)
- `versionx-lockfile`
- `versionx-state` (rusqlite + WAL, minimal schema — `repos`, `runtimes_installed`, `runs`)
- `versionx-events` (tokio broadcast bus)
- `versionx-shim` (Rust trampoline; Unix symlink, Windows copy-per-tool)
- `versionx-runtime-trait` + `versionx-runtime-node`
- `versionx-adapter-trait` + `versionx-adapter-node` (pnpm primary; npm/yarn stubs)

**Commands:**
- `versionx init` — synthesize `versionx.toml` from `package.json`
- `versionx sync` — install Node + pnpm + run `pnpm install --frozen-lockfile`
- `versionx verify` — check lockfile integrity
- `versionx install <tool> <version>` — global runtime install
- `versionx global set <tool> <version>` — user-default pin
- `versionx which <tool>` — show resolved binary + why
- `versionx exec --runtime node@X.Y.Z -- ...` — one-off version override
- `versionx activate {bash|zsh|fish|pwsh}` — emit shell hook
- `versionx --version`, `versionx self-check`, `versionx --help-json`

**Platform:**
- Shell hooks for bash/zsh/fish/PowerShell
- XDG-compliant paths on Linux; platform-correct on macOS/Windows
- Volta-style Windows trampoline with mmap'd PATH cache
- `.env` / `.env.local` auto-loading
- `.nvmrc` and `package.json#engines.node` read-compat

**Demo (30s):**
```bash
curl -fsSL https://versionx.dev/install.sh | sh
cd my-pnpm-repo   # no versionx.toml
versionx init            # auto-detects Node + pnpm
versionx sync            # installs Node, installs pnpm, runs pnpm install
versionx which pnpm      # shows resolved version
```

### Tool version matrix (CI)
- Node 18, 20, 22 LTS
- pnpm 8, 9, 10
- npm 9, 10
- yarn 1, 4

### Known out-of-scope (explicit)
Python. Rust. Daemon (direct mode only). Release engine. Policy. MCP. Cross-repo.

## 0.4.0 — Release engine, single-repo (+2.5 months)

**Target date: M+8**. The wedge starts here.

### Scope
Plan/apply pipeline, four release strategies, polyglot single-repo release, OIDC publishing.

**New crates:**
- `versionx-release` (PR-title parser, Conventional Commits parser, changeset file handling, SemVer + CalVer versioners, cross-ecosystem version translator)

**Features:**
- **PR-title strategy as default** (squash-merge friendly).
- Conventional Commits, Changesets, Manual strategies.
- Plan/apply with Blake3 IDs + `pre_requisite_hash` + configurable TTL (1h–30d, 24h default).
- Changesets: `versionx changeset add`, `versionx changeset check`, snapshot validation.
- Cross-ecosystem version translation (`-rc.1` ↔ `rc1` for PEP 440, caret-behavior differences flagged).
- Monorepo per-package releases + dependency graph cascade.
- Topological publish with deterministic tie-breaking.
- `versionx release propose`, `versionx release show`, `versionx release approve`, `versionx release apply`.
- Interactive push prompt in TTY; `--push`/`--no-push` required in CI.
- OIDC trusted publishing: npm, PyPI, crates.io, RubyGems (detection + fallback with loud warning).
- `versionx release snapshot --tag canary` with registry-regex pre-validation.
- Changelog generation via git-cliff skeleton.

**Commands (new):**
- `versionx release {propose,show,approve,apply,rollback,snapshot}`
- `versionx changeset {add,check,list}`
- `versionx release prerelease {enter,exit}`
- `versionx plan {list,show,apply,expire}`

**Demo (60s):**
```bash
versionx changeset add          # interactive TUI
git commit && git push
# PR merged to main:
versionx release apply          # publishes Node pkg to npm (OIDC), Python pkg to PyPI, Rust crate to crates.io
```

### Gates beyond baseline
- [ ] Cross-ecosystem release E2E test (verdaccio + local PyPI + local cargo registry).
- [ ] OIDC publish E2E test with GitHub Actions + real trusted publishing config.
- [ ] Plan TTL enforcement tested (expired plans refused).
- [ ] Rollback E2E: release, rollback, verify tags/commits reverted.

## 0.5.0 — Policy engine + waivers (+1.5 months)

**Target date: M+9.5**. Enforceable gates.

### Scope
Declarative rules, Luau escape hatch, mandatory-expiry waivers, policy lockfile.

**New crates:**
- `versionx-policy` (TOML parser, Luau runtime via mlua, rule evaluators, waiver resolver)

**Features:**
- All 10 built-in rule types: `runtime_version`, `dependency_version`, `dependency_presence`, `advisory_block`, `release_gate`, `commit_format`, `lockfile_integrity`, `link_freshness`, `provenance_required`, `custom` (Luau).
- Luau sandbox: `io`/`os`/`package`/`debug` stripped; `set_interrupt` (100ms default); `set_memory_limit` (32MB default); http.get rate-limited + cached.
- Severity (`deny`/`warn`/`info`), scope (all/tag/path/list), triggers.
- Waivers with **mandatory `expires_at`** by default; org override; 7-day pre-expiry warnings.
- `versionx.policy.lock` — inherited policies pinned to content SHA.
- Sealed policies cannot be disabled downstream.
- Evaluation cache in `state.policy_evaluations`.

**Commands (new):**
- `versionx policy {init,add,test,check,explain,verify,update,stats,audit}`
- `versionx waiver {add,list,audit,expire}`

**Demo (30s):**
```bash
versionx policy add runtime_version --runtime node --min 20
versionx release propose     # FAILS: "Node 18 below minimum 20"
versionx waiver add --policy runtime:node --reason "Legacy app, Q2 migration" --expires 2026-06-01
versionx release propose     # passes with waiver noted
```

### Gates beyond baseline
- [ ] Luau sandbox fuzzed (can't escape via upvalue tricks, metatables, or coroutines).
- [ ] mlua version pinned + audited; advisories watched.
- [ ] Policy-lockfile drift test: central update doesn't break members until explicit `versionx policy update`.

## 0.6.0 — MCP server + AI overlay (+1.5 months)

**Target date: M+11**. AI-as-client lands.

### Scope
Full MCP server, BYO-API-key path, voice-aware changelog pipeline.

**New crates:**
- `versionx-mcp` (rmcp-based, stdio + loopback HTTP)

**Features:**
- Built on official `rmcp` Rust SDK ≥1.5.
- Transports: stdio (primary) + loopback HTTP (no auth).
- **≤10 tools**, workflow-shaped, with `_plan` / `_apply` variants where mutating.
- Read-only resources (`versionx://config`, `versionx://state/repos`, etc.) **mirrored as tools** for clients that don't surface resources.
- 3 preloaded prompts: `propose_release`, `audit_dependency_freshness`, `remediate_policy_violation`.
- Elicitation as progressive enhancement (works on Claude Code/VS Code; graceful fallback for Cursor/Cline).
- Structured + text content on every response; `resource_link` for large payloads.
- Tool output sanitization for prompt-injection defense (untrusted content in fenced blocks with markers).
- BYO-API-key path: `[release.ai.byo] provider = "anthropic|openai|gemini|ollama"` + key env var.
- Voice-aware changelog: state-DB `changelog_voice` table captures past samples; MCP tool context includes README + CHANGELOG + last-10-samples; agent generates prose; `versionx_write` tool commits draft.
- Per-call audit log with `clientInfo` (spoofable, logged only).

**Commands (new):**
- `versionx mcp serve [--stdio|--http PORT]`
- `versionx ai configure` — interactive wizard for MCP client setup or BYO key
- `versionx changelog draft` — regenerate voice-aware draft using configured provider

**Demo (60s):**
```bash
# in Claude Code, ".mcp.json" already points to versionx mcp serve --stdio
# Claude Code, chat:
> /versionx__propose_release
# Claude reads repo, generates voice-aware changelog, proposes bumps
# User clicks approve in elicitation dialog
# versionx release apply runs
```

### Gates beyond baseline
- [ ] Integration test against Claude Code, Cursor, Codex actual clients (not mocks).
- [ ] Tool output injection-attack test suite (70+ payloads from MCPTox benchmark).
- [ ] BYO-API-key path works against all four providers with recorded fixtures.

## 0.7.0 — Cross-repo coordination (+2 months)

**Target date: M+13**. The wedge fully lands.

### Scope
Virtual monorepo, cross-repo release sets, saga rollback.

**New crates:**
- `versionx-multirepo` (submodule/subtree/virtual/ref link handlers)
- `versionx-git` (gix for reads, git2 for writes)

**Features:**
- `versionx-fleet.toml` in dedicated ops repo pattern.
- `versionx fleet {init,sync,status,query,members}` — parallel operations across N repos.
- `versionx links {sync,check-updates,update,pull,push}` for submodule + subtree + ref.
- Cross-repo release sets in 3 modes:
  - `independent`: parallel, no atomicity.
  - `gated`: explicit dependency order, stop on fail.
  - `coordinated`: saga pattern with interactive rollback.
- Saga protocol: dry-run all → tag all locally → publish in topo order → on fail, prompt for (manual rescue / auto-revert commits / best-effort yank).
- Cross-ecosystem + cross-repo version translation with lossy-conversion warnings.
- `refs/versionx/history` git-backed run history for state-DB recovery.

**Commands (new):**
- `versionx fleet release {propose,apply,show,rollback}`
- `versionx state {backup,restore,repair}`

**Demo (90s):**
```bash
cd platform-ops && cat versionx-fleet.toml   # 3 repos in customer-portal set
versionx fleet release propose --set customer-portal
# Shows: frontend 1.4 → 1.5, api 2.0 → 2.1, shared 0.9 → 1.0
versionx fleet release apply
# Dry-run all → tag all → publish frontend → publish api → FAIL publishing shared
# Prompt: manual rescue / auto-revert / yank?
# User picks auto-revert. versionx unrolls all three publishes.
```

### Gates beyond baseline
- [ ] Saga rollback E2E: 3-repo release with intentional mid-way failure; all three rollback pathways tested.
- [ ] Cross-ecosystem version translation tested through npm + PyPI + crates.io in a single set.
- [ ] `refs/versionx/history` recovery tested (delete state.db; restore; verify).

## Dependencies between releases

```
0.1 (Node)  ─────────────────────────────────────► all later
0.2 (Py+Rs) ─► 0.3 daemon (polyglot is more useful with fast re-runs)
0.3 daemon  ─► 0.4 release (streaming release progress needs daemon RPC)
0.4 release ─► 0.5 policy (policy needs release context to gate)
0.5 policy  ─► 0.6 MCP (MCP tools surface plan/apply + policy)
0.6 MCP     ─► 0.7 cross-repo (AI-assisted cross-repo needs MCP stability)
0.7 cross   ─► 0.8 Actions (Actions use cross-repo semantics)
0.8 Actions ─► 0.9 task runner (CI demos lean on tasks)
0.9 tasks   ─► 1.0 GA
```

Non-blocking parallelism opportunities:
- Docs writing can track any version +1 ahead.
- CI/build infra for 0.1 is reusable forever.
- Fixture repos for 0.2 inform 0.4 (same monorepo reused).

## Exit criteria for "complete production"

1.0 is **not** "complete" — it's "GA and semver-committed." Versionx reaches what a reasonable person calls **complete production** at around **1.4**, when:

1. All Tier 1 + Tier 2 ecosystems are supported (Node, Python, Rust, Go, Ruby, JVM, OCI).
2. Remote state + web UI + full task runner with remote cache are shipped.
3. Supply-chain depth (OIDC, SLSA, sigstore, in-toto) is integrated end-to-end.
4. Real-world production case studies exist for all 5 signal-of-success personas.
5. Monthly downloads are in the tens-of-thousands range (adoption signal without telemetry).
6. A security audit has passed with no unresolved high-severity findings.
